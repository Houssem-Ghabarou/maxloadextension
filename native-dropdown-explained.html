How MaxLoad reliably sets the Maximo status

The “dropdown never opens” problem — and the native method that made it disappear.

Short version: instead of simulating a user clicking the status dropdown, we now ask Maximo to change the status using its own internal function. The menu never has to open on screen, so it can’t fail to open.

1. Why the old way kept failing

The Maximo status menu is not a normal <select> box. It’s a custom menu that Maximo builds with its own JavaScript. To change the status the “human” way you must:

  1. Click the little dropdown arrow next to the New Status field…
  2. …wait for Maximo to build and show the menu…
  3. …then click the right option in that menu.

The trap Maximo ignores fake (synthetic) clicks. A browser extension can fire a “click” event in JavaScript, but Maximo checks whether the click is a real one from the mouse/OS (isTrusted). Our synthetic clicks were being thrown away — so the arrow was “clicked,” but the menu never opened. Every later step then failed because the menu it needed wasn’t there.

2. The breakthrough: Maximo tells us how to do it

When you open the Change Status dialog, Maximo puts the whole status menu into the page — even before it’s visible. Looking at one option in the page source, each choice is a link like this:

<a id="menu0_APPR_OPTION_a" role="menuitem"
   href="javascript: topLevelMenus['shared'].menuClick(
     {"text":"Approved","id":"APPR_OPTION","value":"APPR_OPTION",
      "target":"status_menus","event":"click"});">
   <span>Approved</span>
</a>

Two gifts are hidden in there:

  • The option’s internal code is baked into its id: menu0_APPR_OPTION_a → APPR. That code is the same in every language (English “Approved”, French “Approuvé”), so we can find the right option regardless of what language the screen is in.
  • The exact function Maximo runs when you click it: topLevelMenus['shared'].menuClick({…}). If we call that function ourselves, we get the identical result as a real click — without needing the menu to be open.

The idea Find the option by its code, then run Maximo’s own menuClick for it. No arrow-click, no waiting for the menu to appear, no coordinates that can “miss.”

3. The one obstacle — and how we get around it

A browser extension normally runs in an isolated world: it can see and touch the page’s HTML, but it cannot call the page’s own JavaScript functions like topLevelMenus. That’s a security boundary.

The trick: an extension is allowed to inject a second script into the page’s MAIN world — the page’s own JavaScript realm — where topLevelMenus does exist. That injected script is our “bridge.” (Even Playwright/Puppeteer can’t reach page globals like this — only an extension can.)

Isolated world — the normal extension code

menu-select.js decides which status to set (code + known labels)
│ sends a message ▼    window.postMessage

MAIN world — the bridge, inside the page

maximo-native.js receives the message
▼
Finds the option by code → calls topLevelMenus['shared'].menuClick(payload)
│ sends the result back ▲

The two halves talk over window.postMessage — the standard, safe way for these two worlds to pass small messages. Every request has a unique id so replies never get mixed up.

4. What the bridge actually does

Finding the right options (and not the wrong menu)

The page also has a “Select Action” toolbar menu (New Work Order, Change Status, Route Workflow…). It’s built the same way, so early on the bridge grabbed it by mistake. The fix: only accept options that belong to the status menu — either inside a menu tagged target="status_menus", or whose menuClick payload says "target":"status_menus".

// only real status options — never the "Select Action" toolbar menu
ul[target*='status']  →  its <a> menu items
// fallback: an _OPTION link whose payload contains "status_menus"

Selecting — the core move

Given a status code (e.g. INPRG), the bridge finds the matching option and runs Maximo’s own click handler:

function invokeOption(a) {
  // read the exact payload Maximo embedded in the link
  const payload = JSON.parse( /menuClick\((\{.*?\})\)/.exec(a.href)[1] );
  // call Maximo's own function — same as a real click, no menu needed
  window.topLevelMenus.shared.menuClick(payload);
}

If for some reason topLevelMenus isn’t reachable, it falls back to clicking the link, which runs the same javascript: code.

Opening — only if the menu isn’t built yet

Some screens build the menu lazily (only when you first open it). If the bridge doesn’t find the status options, it locates the New Status field — by the id we captured while teaching, or by its label (“New Status” / “Nouveau statut”) — finds its arrow, triggers Maximo’s native open, waits briefly, then selects. In your case the menu is already built when the dialog opens, so this step is usually skipped.

5. The full journey of one status step

  1. During a run, the status step asks: “set status to INPRG.”
  2. The isolated code turns your value into a code + known labels and messages the bridge in every frame.
  3. The frame that owns the status menu answers; the others say “not here.”
  4. The bridge finds the INPRG option and runs Maximo’s menuClick.
  5. Maximo sets the status itself. The field updates. Done — the dropdown never had to open.

If the native path can’t run (a screen with no topLevelMenus), it quietly falls back to the old trusted-click method, so nothing breaks.

6. How you tested it

You talked to the bridge straight from the browser console. The message asks a question; the bridge answers.

Ask what it can see (safe — changes nothing):

new Promise(r=>{const id="t"+Math.random();
addEventListener("message",function h(e){const d=e.data;
  if(d&&d.__mlNative&&d.dir==="res"&&d.id===id){removeEventListener("message",h);r(d)}});
postMessage({__mlNative:true,dir:"req",id,op:"probe",args:{}},"*")})

Actually set one: the same message with op:"setStatus" and args:{code:"INPRG", …} → returned {ok:true, code:"INPRG", label:"In progress"}, and the field changed on the real screen. ✅
