Architecture note

MaxLoad — Chrome Extension vs. Standalone Desktop App

Same engine, two delivery vehicles. What actually changes when MaxLoad runs as an Electron desktop app instead of a browser extension.

Bottom line: Per-row automation speed is essentially identical — same Chromium, same engine, same CDP input, and the real bottleneck (Maximo's own server round-trips) is unchanged. Long-run reliability is likely better on desktop. The costs are higher memory, a bigger install, a slightly slower cold start, and one current coverage gap (sub-frame injection).

⚡Why speed is the same

The factors that decide how fast a row processes don't change between the two:

  • Same rendering engine. Electron embeds Chromium — Maximo renders exactly as in Chrome (same DOM, same Dojo).
  • The automation core runs identically. The matcher, DOM analyzer, settle detector, recorder, and error-watcher still run as JavaScript inside the Maximo page — byte-for-byte the same code.
  • CDP input is the same protocol. The extension used chrome.debugger → CDP; the desktop app uses webContents.debugger → CDP. Same trusted Input.dispatchKeyEvent keystrokes.
  • Maximo is the bottleneck. Every field waits on Maximo's server round-trips and settle detection. That dominates timing by orders of magnitude — our messaging overhead is noise beside it.

✅Where the desktop app is actually better

No more service-worker eviction

Manifest V3 service workers are killed after ~30s idle. In the extension, the worker that owns CDP input and the AI proxy could be evicted mid-run and must respawn — a known source of flakiness on long/overnight batches. In the desktop app the main process never sleeps; it stays alive for the entire run.

Isolated browser

A dedicated Chromium for automation — unaffected by the user's other tabs, extensions, or an accidental tab-close.

No scary permission

CDP attaches to our own view. No Chrome "a debugger is attached" banner, no <all_urls> extension review to pass.

No platform risk

Independent of Google's extension-policy changes (MV3 already broke a lot). You control the runtime.

Ownable & licensable

A signed installed app IT can vet — and the natural place to enforce a server-side license + AI key.

🔐Login — unchanged, and safe

We do not automate or store the Maximo login. The app embeds a real Chromium view; the user types the Maximo URL and logs in manually — password, SSO, or MFA all work as in a normal browser. The session persists in a named partition, so it's usually a one-time sign-in. MaxLoad only ever acts on that live, already-authenticated session — the same assumption the extension made about your open Maximo tab.

🚧The one real gap today

Sub-frame injection (coverage, not speed)

The prototype injects content scripts into the main frame only. The DOM analyzer still recurses same-origin iframes from the top frame, so most Maximo forms are reachable — but the always-on error-watcher does not yet run inside each child frame the way the extension's all_frames does. On iframe-heavy screens it could miss a modal. This is the next step to reach true 1:1 parity.
